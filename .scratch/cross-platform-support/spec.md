# Linux/Windows 可移植架构与进程详情

Status: ready-for-agent

## Problem Statement

Delray 当前面向 Linux `x86_64`，但主要技术栈具备 Windows 可移植性。实际阻碍集中在 Linux `/proc` 进程查询、固定 Ethernet 数据链路解析、Windows Npcap 构建与运行依赖，以及缺少 Windows 原生环境验证。

跨平台改造不能只做到编译通过。Windows 上的网卡和 IP 统计、进程归属、TUI、plain、JSON/JSONL 必须实际可用；进程查询失败时仍需保持「未归属流量」语义，不能让进程统计整体失效。与此同时，项目不应为了维持应用程序的低 MSRV 而自行重写成熟库已经提供的平台能力。

当前主力开发环境是 Linux，暂不具备 Windows CI。工作需要先在 Linux 完成共享改造，再在独立的短期 Windows 分支和 Windows `x86_64` 真机上完成构建、修正与人工验收。macOS 没有真机环境，不属于本规格的支持范围。

## Solution

Delray 建立可移植架构，将进程查询、数据链路解析、统计和展示组织为清晰模块。进程查询使用 `listeners 0.6`，MSRV 提高到 Rust 1.88；Linux 与 Windows 共用统一的 socket 索引、进程统计身份和失败语义。抓包解析根据 pcap link type 支持 Ethernet、Raw IP、NULL/LOOP、Linux SLL 和 SLL2。

Linux `x86_64` 继续作为支持平台。Windows `x86_64` 在开发和人工验收完成前是目标平台，验收通过后再成为支持平台。Windows 使用者自行从 Npcap 官方网站安装 Npcap；Delray 不捆绑、不下载、不分发 Npcap。

进程数据增加可执行文件路径和 `Last seen`。概览页和进程列表保持紧凑；从进程列表进入单进程详情视图后展示路径、流量和新鲜度。进程离开 top-N 时详情不自动退出，而是保留最后数据并显示英文 `Tracking paused` 状态。plain 和 JSON/JSONL 同步增加路径与最后流量时间。

## User Stories

1. 作为 Linux 运维人员，希望升级后现有抓包、进程统计和输出行为继续可用，以便跨平台改造不造成 Linux 回归。
2. 作为 Windows 使用者，希望能够在 Windows `x86_64` 上从源码编译出可运行的 `delray.exe`，以便分析 Windows 主机流量。
3. 作为 Windows 使用者，希望 README 提供 Npcap 官方网站和必要的构建、运行说明，以便自行安装依赖。
4. 作为 Windows 普通用户，希望安装 Npcap 后能够启动 Delray，而不是被强制要求管理员权限。
5. 作为 Windows 管理员，希望提升权限后能够获得更完整的系统进程归属，以便分析普通用户无权读取的进程。
6. 作为使用者，希望缺少 Npcap 或抓包权限时看到明确的英文错误，以便知道应安装依赖还是调整权限。
7. 作为交互式使用者，希望无网卡参数时看到带数字编号、友好描述和底层设备名的网卡列表，以便选择长名称的 Windows Npcap 设备。
8. 作为交互式使用者，希望位置参数可以接受当次网卡列表中的数字编号，以便避免复制 Npcap GUID。
9. 作为脚本编写者，希望位置参数继续接受完整 pcap 设备名，以便使用不依赖临时编号的稳定命令。
10. 作为 Linux 使用者，希望能够抓取物理 Ethernet 网卡，以便保持现有主要使用方式。
11. 作为 Windows 使用者，希望能够抓取 Npcap Loopback，以便分析本机服务之间的流量。
12. 作为 Linux 使用者，希望能够抓取 loopback 和 `any` 设备，以便分析本机和聚合流量。
13. 作为使用者，希望 Delray 能够解析 Raw IPv4 和 IPv6 数据链路，以便支持没有 Ethernet 头的捕获设备。
14. 作为使用者，希望 Delray 能够解析 NULL/LOOP、Linux SLL 和 SLL2，以便相应设备不会静默产生空统计。
15. 作为使用者，希望打开未知数据链路类型时立即看到明确错误，以便不会误以为没有网络流量。
16. 作为流量分析人员，希望 socket 查询同时考虑本机 IP、端口和 TCP/UDP 协议，以便避免同端口协议冲突导致错误归属。
17. 作为流量分析人员，希望具体本机地址优先于 `0.0.0.0` 或 `::` 通配监听，以便流量归属到更精确的候选进程。
18. 作为流量分析人员，希望只有唯一候选 PID 时才归属流量，以便共享 socket 不会被任意分配给第一个进程。
19. 作为流量分析人员，希望没有候选进程或存在多个候选进程时都计入「未归属流量」，以便总流量不丢失且不制造错误归属。
20. 作为流量分析人员，希望已知 PID 但路径缺失时仍保留进程归属，以便权限不足不会抹掉已经确定的身份。
21. 作为流量分析人员，希望相同 PID 对应不同可执行路径时拆分统计身份，以便 PID 复用或 `exec` 不会混合两个程序的历史流量。
22. 作为流量分析人员，希望路径不可用时仍可按 PID 累计，以便部分信息缺失不阻断进程统计。
23. 作为流量分析人员，希望进程显示名统一为稳定的可执行文件名，以便 Linux 和 Windows 具有一致语义。
24. 作为流量分析人员，希望完整命令行不混入进程显示名，以便启动参数变化不会造成名称不稳定。
25. 作为流量分析人员，希望概览页只显示进程名和总流量，以便有限空间保持可扫读。
26. 作为流量分析人员，希望进程列表显示进程名、PID、接收、发送和总流量，以便快速比较 top-N 进程。
27. 作为流量分析人员，希望在进程列表中选中一行并按 `Enter` 进入详情，以便查看该进程的完整信息。
28. 作为流量分析人员，希望进程详情显示进程名、PID、可执行路径、接收、发送、总流量和 `Last seen`，以便判断流量来源和数据新鲜度。
29. 作为流量分析人员，希望未归属流量也可以进入详情，以便查看其累计流量和 `Last seen`；其 PID 和路径显示为 `-`。
30. 作为流量分析人员，希望 `Last seen` 表示最后一次实际捕获并归属到该统计身份的流量时间，以便进程表刷新不会伪造活动时间。
31. 作为流量分析人员，希望进程路径或名称变化不更新 `Last seen`，以便该字段只表示流量活动。
32. 作为流量分析人员，希望详情中的进程离开当前 top-N 后页面不自动退出，以便有时间阅读最后数据。
33. 作为流量分析人员，希望进程离开 top-N 后看到 `Last seen` 和一次性英文提示，以便知道数据不再实时更新。
34. 作为流量分析人员，希望非最新详情显示 `Tracking paused: process is no longer in Top-N.`，以便状态不会被误解为进程已经退出。
35. 作为流量分析人员，希望进程重新进入 top-N 后详情恢复实时更新，以便无需退出并重新选择。
36. 作为流量分析人员，希望进程查询数据过期时看到 `Tracking paused: process data is stale.`，以便区分排名变化和查询故障。
37. 作为文件输出使用者，希望 plain 进程记录增加 `Path` 和 `Last Seen`，以便离线报告包含进程位置和新鲜度。
38. 作为 JSON/JSONL 消费者，希望进程记录增加可空的 `path` 和 RFC 3339 格式的 `last_seen`，以便程序可以稳定解析。
39. 作为 JSON/JSONL 消费者，希望未归属流量使用 `pid: null`、`name: null`、`path: null`，以便特殊身份不伪装成普通进程。
40. 作为统计使用者，希望未归属流量继续正常参与 top-N 排序，以便不会被固定置顶或额外增加一行。
41. 作为统计使用者，希望每个流量样本只进入一个真实进程统计身份或未归属流量，以便进程维度和接口维度保持分区不变量。
42. 作为统计使用者，希望后续成功查询到进程时不追溯修改历史流量，以便累计统计保持可解释。
43. 作为使用者，希望一次瞬时 `listeners` 刷新失败不会立即清空进程表，以便避免未归属流量突然增加。
44. 作为使用者，希望旧进程表最多使用 `2 × --proc-refresh`，以便短暂错误可恢复而长期错误不会造成陈旧 PID 误归属。
45. 作为使用者，希望旧进程表过期后新流量进入未归属流量且抓包继续，以便查询故障不终止主功能。
46. 作为开发者，希望 `listeners` 被封装在项目进程查询接口后，以便统计与展示代码不依赖第三方数据模型。
47. 作为开发者，希望使用 `listeners::get_all()` 建立索引，而不是按端口查询，以便保留地址和协议并避免端口歧义。
48. 作为开发者，希望 Rust 1.88 是可验证的最低编译器版本，以便依赖需求明确且不会为 Rust 1.85 重写系统能力。
49. 作为 Linux 二进制使用者，希望 glibc 2.28 和 libpcap 运行基线保持不变，以便提高 MSRV 不影响目标机兼容性。
50. 作为开发者，希望所有用户可见 TUI、plain、JSON 字段和状态文案保持英文，以便产品语言一致。
51. 作为维护者，希望共享改造先在短期 `cross-platform-core` 分支完成并通过 Linux 验证，以便主力环境能够快速反馈。
52. 作为 Windows 实施人员，希望从最新主干创建短期 `windows-support` 分支，以便在 Windows 真机集中完成平台修正。
53. 作为维护者，希望 `main` 始终是唯一集成主线，以便 Linux 和 Windows 不形成长期分叉。
54. 作为维护者，希望 Windows 发现的共享问题回到共享模块，以便平台分支不复制业务逻辑。
55. 作为维护者，希望 Windows 在人工验收前仍称为目标平台，以便项目不提前宣称支持。
56. 作为维护者，希望 Windows 人工验收通过后再更新 README 的支持平台和重点说明，以便文档与事实一致。
57. 作为 Windows 实施人员，希望能够记录 Windows、Rust、Npcap 版本、命令和测试结果，以便人工验收可以复现。
58. 作为项目维护者，希望本轮只要求生成可用 Linux 和 Windows 构建产物，以便不被安装包、签名和正式发布工程阻塞。
59. 作为流量分析人员，希望 PID 已知但进程名或路径缺失时仍保留进程归属，以便部分身份信息缺失不会把已确定的 PID 变成未归属流量。
60. 作为开发者，希望进程归属诊断能够量化 lookup 未命中、刷新耗时和未归属原因，以便区分实现竞态、权限限制和系统边界。
61. 作为流量分析人员，希望 socket lookup 未命中时触发合并且限速的进程表刷新，以便减少刷新间隔内新连接被永久计入未归属流量的概率。
62. 作为维护者，希望任何短窗口待归属缓冲先通过独立决策修改历史流量语义，以便不会隐式破坏当前累计统计解释。

## Implementation Decisions

- Linux `x86_64` 是当前支持平台。Windows `x86_64` 是目标平台，只有完成 Windows 原生编译和人工功能验收后才能成为支持平台。
- macOS 不进入构建、测试或可用性承诺。架构应允许未来增加平台实现，但不得以可移植架构推断 macOS 已受支持。
- ARM64、x86 和其他 CPU 架构不进入本轮验收。
- 使用 `listeners 0.6` 作为 Linux 与 Windows 的进程查询基础，删除项目自行维护的 Linux `/proc` socket 到 PID 实现。
- MSRV 从 Rust 1.85 提高到 Rust 1.88。项目锁定依赖解析结果，并使用 Rust 1.88 验证最低编译器版本。提高 MSRV 不改变 Linux glibc 2.28 和 libpcap 运行基线。
- `listeners` 放在项目内部进程查询模块后；统计、管线、报告和 TUI 不直接依赖第三方类型。
- 进程查询模块使用 `listeners::get_all()` 周期性建立索引，不使用 `get_process_by_port()`。
- socket 查询键包含本机 IP、端口和 TCP/UDP 协议。数据包解析结果必须携带协议，不能再以 `(IP, port)` 表示 socket。
- 具体本机地址优先于同协议、同端口的 IPv4 或 IPv6 通配地址。最终候选只有一个 PID 时返回该进程；不同 PID 的多个候选返回歧义并计入未归属流量。
- `listeners` 刷新失败时保留最后一次成功快照，最大有效期为 `2 × --proc-refresh`。过期后停止使用旧快照；查询错误写入 stderr 或日志，但不终止抓包。
- 后续刷新成功后恢复进程查询，不追溯重分配旧流量。
- 进程归属应暴露可测试诊断数据，用于量化未命中、刷新请求、刷新耗时和未归属原因；诊断不得阻塞抓包、聚合或 TUI 事件循环。
- socket lookup 未命中可以请求进程表提前刷新，但刷新请求必须合并、限速并在进程表线程中执行；聚合线程不得同步扫描系统进程表。
- 短窗口待归属缓冲会改变「后续成功查询不追溯历史流量」语义，必须通过独立 ticket 和规格/ADR 决策后再实现。
- 进程显示名使用可执行文件名，不使用完整命令行。进程名和可执行路径均为可空身份字段；PID 已知时不能因名称或路径缺失而变成未归属流量。缺失名称沿用现有未知名称表示，缺失路径使用 `-` 或 `null`。
- 进程统计身份以 PID 为基础；路径可用时以 PID 和路径共同区分。PID 相同而路径变化时创建新的统计身份；路径缺失时按 PID 区分。
- 每个进程统计身份和未归属流量记录最后一次实际流量时间。进程表刷新、名称变化和路径变化不更新该时间。
- `Last seen` 在 TUI 中显示相对时间，在 plain 中使用 ISO 8601，在 JSON/JSONL 中使用 RFC 3339。
- pcap 捕获打开后保存实际 link type。解析模块支持 Ethernet、Raw IPv4/IPv6、NULL/LOOP、Linux SLL/SLL2；不支持的 link type 在打开或首次确定类型时返回明确错误，不能静默丢弃全部包。
- 无网卡参数时使用英文编号列表显示友好描述和完整 pcap 名称。位置参数接受完整设备名或当次列表的数字编号。编号不承诺跨进程稳定，自动化脚本使用完整名称。
- 默认路由标记属于尽力而为。非 Linux 平台不要求实现默认路由检测。
- TUI 概览页和进程列表不展示路径。进程列表支持明确的选中状态，按 `Enter` 进入单进程详情，并提供明确的返回操作。
- 进程详情保存所选进程统计身份和最后一份详情数据。新快照仍包含该身份时更新数据；离开 top-N 时保留最后数据，不自动退出。
- 离开 top-N 的状态前缀使用 `Tracking paused`，显示一次 `Tracking paused: process is no longer in Top-N.` 提示，并持续显示相对 `Last seen`。该身份重新进入 top-N 后恢复实时状态。
- 进程查询快照过期导致详情无法继续更新时使用 `Tracking paused: process data is stale.`。文案不得声称进程已经退出，因为系统没有可靠的进程退出证据。
- 未归属流量允许进入详情；PID 和路径使用 `-`，名称继续使用特殊标签和现有特殊样式。
- plain 进程表在现有字段后增加 `Path` 和 `Last Seen`。缺失路径使用 `-`。
- JSON/JSONL 进程记录增加 `path` 和 `last_seen`。缺失身份或路径使用 `null`；时间使用 RFC 3339 字符串。
- TUI、plain、JSON 字段说明、CLI 提示和错误全部使用英文。中文只用于项目技术文档、研究记录和领域词汇。
- Windows 运行时依赖使用者单独安装 Npcap。项目不捆绑、不自动下载、不再分发 Npcap；最终 README 链接 `https://npcap.com/` 并说明 Npcap SDK、运行时和权限重点。
- Windows 普通用户可以运行。管理员权限只用于提高受保护进程的归属完整度，不作为程序启动前置条件。
- 开发采用两个顺序执行的短期分支：先在 Linux 完成并合并共享改造，再从最新主干创建 Windows 分支进行真机修正。不得保留长期 Linux/Windows 产品分支。
- 开发期间不修改 README 的支持平台声明。Windows 人工验收完成后再补充支持平台、Npcap 官方链接、Windows 构建和运行重点。
- 当前没有 Windows CI。本轮 Windows 证据来自 Windows `x86_64` 真机的人工编译与测试，并保留环境版本、命令和结果记录。
- 本轮只要求生成可用构建产物，不要求 ZIP、MSI、代码签名、自动发布或正式版本分发。

## Testing Decisions

- 自动化测试只验证模块接口可观察行为，不断言 `listeners`、pcap 或操作系统后端的内部实现。
- 数据链路解析 seam 接受 link type、固定字节样本和本机 IP，返回流量记录。测试覆盖 Ethernet、Raw IPv4/IPv6、NULL/LOOP、Linux SLL/SLL2、TCP、UDP、IPv4、IPv6、非本机流量和未知 link type。
- 进程查询 seam 使用可注入的 `listeners` 记录、刷新结果和可控时间。测试覆盖具体地址、IPv4/IPv6 通配地址、TCP/UDP 同端口、唯一 PID、重复同 PID、多个不同 PID、名称缺失、路径缺失、路径变化、刷新错误、有效旧快照、过期旧快照和恢复刷新。
- 进程归属诊断和刷新调度使用可注入时间、查询函数和唤醒通道测试。测试覆盖首次命中、未命中、表过期、刷新合并、刷新限速、刷新失败和聚合线程非阻塞行为。
- 统计 seam 沿用从流量记录到不可变统计快照的现有接口。测试覆盖 PID + 路径身份拆分、显示名、路径、`last_seen`、双方向流量、未归属流量、正常 top-N 竞争、零流量空状态和接口流量分区不变量。
- 展示 seam 沿用 plain/JSON 纯渲染与 ratatui `TestBackend`。测试覆盖新增字段、空值、RFC 3339、英文文案、进程选择、进入/退出详情、路径展示、相对 `Last seen`、离开 top-N 后保留数据、一次性提示、持续 paused 状态和重新进入 top-N 后恢复。
- 继续运行现有 CLI、管线、TUI 延迟和输出测试，防止 Linux 行为、事件循环响应和未归属流量语义回归。

## Out of Scope

- macOS 构建、测试、真机验证和可用性承诺。
- Linux 或 Windows ARM64、Windows x86 及其他 CPU 架构。
- Windows CI、自动化 Npcap 实时抓包环境和自动发布工作流。
- ZIP、MSI、DEB、RPM 等新增打包工作。
- Windows 代码签名、证书管理、SmartScreen 信誉和安装器。
- Npcap 捆绑、自动下载、OEM 再分发和授权采购。
- 正式版本发布、发布页面和二进制分发策略。
- 在概览页或进程列表中展示完整路径。
- 获取或展示完整进程命令行。
- 可靠判断进程已经退出；详情 paused 状态只表示数据不再更新。
- 进程离开 top-N 后继续在后台获取不属于 top-N 的实时统计。
- 后续识别到进程时追溯修改已经计入未归属流量的历史数据。
- 保证 Linux 与 Windows 达到完全相同的进程归属率。
- 为非 Linux 平台实现默认路由识别。
- 长期维护独立 Linux 和 Windows 产品分支。

## Further Notes

- 已接受 ADR「使用 listeners 并将 MSRV 提高到 Rust 1.88」。
- `listeners 0.6.0` 已在最小工程中验证：Rust 1.85 编译失败，Rust 1.88 编译通过。
- 参考项目 Sniffnet 证明 `pcap`、`etherparse` 和 `listeners` 的组合可以在 Linux 与 Windows 使用，但 Delray 需要保持自己的未归属流量、top-N、进程统计身份和不追溯语义，不能直接复制 Sniffnet 的端口查询或历史重归属行为。
- 本规格应在下一阶段拆为阻塞关系明确的 tracer-bullet tickets：共享模型与依赖基线、数据链路解析、进程查询与统计、输出与详情页、Linux 验收、Windows 分支与人工验收。
