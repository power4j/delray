# TUI 调色板按终端色深降级，并提供会话级设置浮层

Delray 的 TUI 原先全部使用 24-bit `Color::Rgb` 硬编码配色（`src/tui.rs` 中 15 个命名常量，约 89 处引用）。在仅支持 16 色或单色的终端（`TERM=linux` 控制台、极简终端、部分 SSH 会话）上，真彩色转义序列被忽略或粗略降采样，深底浅字的自定义主题常退化为不可读。Delray 改为按终端色深提供三档语义调色板：真彩色（保持现有 RGB 不变）、16 色（手挑 ANSI 基色映射，底色改用 `Color::Reset` 跟随终端配色，靠 `Modifier` 制造层次）、单色（完全不用颜色，仅靠 `Modifier` 与文本符号区分）。档位在启动时按 `NO_COLOR`、`COLORTERM`、`TERM` 启发式检测一次；crossterm 无现成色深 API，故用环境变量启发式，不引入额外 crate。

激活的调色板通过模块级 `static` 变量保存，由一组访问器函数（如 `accent()`）读取。这避免了把调色板逐层穿透进各 draw 函数——约 89 处调用点只需把常量名机械替换为访问器调用，是改动最小的接缝。该 `static` 仅作单线程事件循环内按键处理与渲染之间的进程内通信，进程退出即失效，不构成持久化。

用户可在设置浮层中手动选择 `Auto`（沿用检测结果）或强制指定某档；选择仅本次会话生效，不写入任何配置文件。设置以浮层形式呈现（小写 `o` 打开、`Esc` 关闭），而非新增标签页——设置不是功能界面，不应进入标签切换循环，且与现有 `i` 打开网卡选择浮层的模式一致。浮层内键位沿用全 App 既有的 `hjkl` 与方向键，不引入第二套方向键体系：同一软件若存在两套方向键（尤其 `k` 在表格中为上、在设置中却为下）会造成肌肉记忆冲突。

16 色档放弃自定义深底改用 `Color::Reset`，因为用户终端配色千差万别，固定深底在 16 色下最易与终端前景色碰撞导致不可读，跟随终端配色最稳。色深档位设计为可注入，使现有 `TestBackend` 测试能钉死真彩色档以保持断言不变，并补充 16 色与单色档的快照测试。
