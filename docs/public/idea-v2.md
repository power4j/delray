# delray 产品定义 v2

> 本文档定义 delray 的产品范围。本版（v2）取代并废弃了初版构想。

## 定位

面向资源受限 Linux 服务器的命令行网络流量分析工具。在服务器出现异常网络流量时，帮助定位流量来源。

## 解决的问题

服务器网络流量突然增大，需要定位消耗原因。难点：

- 异常发生时间不可预知
- 需要定位到具体进程与 IP
- 运行环境资源受限，1 CPU + 1G 内存 + 20G 磁盘为常见配置

现有工具的不足：

| 工具 | 问题 |
|---|---|
| 云监控面板 / Netdata | 只能看整机收发速率，无法定位更细线索 |
| tcpdump | 能抓数据，但无统计与分析能力 |
| vnstat | 提供的数据有限 |
| ntopng | 功能重、安装复杂、资源占用高，目标环境跑不起来 |
| nethogs | 信息够用，但专注实时，缺乏时间段内的统计与分析 |

## 设计原则

- **低资源优先**：可在 1 CPU + 1G 内存环境持续运行
- **CLI 运行**：只需支持 Linux
- **聚合不留存**：维护统计计数，不存储原始报文

## v1 功能范围

### 运行方式

- 支持选择网卡
- 前台运行（默认）：每 5 秒刷新统计数据到终端
- 后台运行（需指定输出文件名）：每 5 秒刷新数据到文件，仅保留最后一次刷新结果

### 输出内容

统计：

- 网卡总流量：按网卡统计的入站与出站总流量
- 进程总流量：按进程统计的接收与发送总流量，取 top 10
- IP 入站总流量：按 IP 统计的入站总流量，取 top 10
- IP 出站总流量：按 IP 统计的出站总流量，取 top 10

运行时信息：开始时间、当前时间、运行时长、当前系统主机名。

## 非目标（v1 不做）

- **HTTP host / TLS SNI 统计**：现代流量以 HTTPS 为主，明文 Host 抓不到，需依赖 SNI 解析；出站可取 SNI，入站仅能从证书反推，覆盖不全且实现复杂度高。v1 暂不实现，纳入后续迭代。
  - 架构上保留对报文 payload 的访问能力（pcap 默认返回完整帧），使该功能可作为增量扩展加入，无需重构。
- 图形界面、非 Linux 平台、实时告警、历史数据持久化。

## 约束

- **资源**：目标在 1 CPU + 1G 内存 + 20G 磁盘环境持续运行
- **权限**：抓包需 root 或 `CAP_NET_RAW`
- **准确性边界**：进程关联基于 `/proc`，短命或已关闭的连接可能漏统计（与 nethogs 等同类工具一致）

## 技术方案

- **语言**：Rust
- **抓包**：`pcap`（libpcap 绑定）。参考 sniffnet 的 buffer（2MB）/ timeout（150ms）参数与 bounded channel 背压策略
- **进程关联**：读取 `/proc/net/tcp`、`/proc/net/udp` 获取 socket inode，关联 `/proc/<pid>/fd` 定位进程。参考 sniffnet 的 `ProgramLookup` 与 nethogs 的实践
- **协议解析**：`etherparse` 或等价库，仅解析必要字段
- **性能**：优先使用 BPF 内核过滤减少进入用户态的报文量；只做聚合计数

## 参考实现

- **sniffnet**（[GyulyVGC/sniffnet](https://github.com/GyulyVGC/sniffnet)）：参考其 pcap 抓包与进程查表部分。其图形界面（egui）与 L7 应用层解析不在本项目参考范围
- **nethogs**：`/proc` 进程关联的成熟实践参考

## 与初版构想的差异

相对初版构想（v1，已废弃）的变更：

1. 明确 HTTP host 统计为非目标，移入后续迭代（backlog）
2. 补充权限、准确性边界、性能策略等约束说明
3. 澄清 sniffnet 的参考边界（仅 pcap 抓包与进程查表，不含 GUI 与 L7 解析）
